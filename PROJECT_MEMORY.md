# Project Memory

## User Intent

Build DawaiSaver.pk as Pakistan's leading medicine intelligence platform, not as a pharmacy website clone.

## Core Product Thesis

The platform should help users upload prescriptions, compare equivalent medicines, optimize prescription costs, discover cheaper alternatives, analyze pharmacy bills, compare pharmacy prices, and eventually connect with pharmacy partners and warehouse fulfillment.

## Critical Constraints

- Governance first.
- Do not implement future phases before previous phase documentation, architecture, and testing are complete.
- Every source must write audit logs.
- Never overwrite historical records.
- Store source URL and source type.
- Every completed task must update AI recovery files and changelog.
- Do not implement frontend, OCR, prescription upload, marketplace, or warehouse work during backend runtime work.

## Preferred Stack

- React, Vite, Tailwind, PWA
- Node.js, NestJS preferred
- Prisma
- PostgreSQL
- BullMQ
- Redis
- Cloudflare R2
- Playwright
- Cheerio

## Database Foundation Memory

- All required Phase 1 tables are modeled in `prisma/schema.prisma`.
- Initial SQL migration is in `prisma/migrations/20260615130000_initial_database_foundation/migration.sql`.
- Every required table includes UUID primary key, soft delete, timestamps, status, source fields, confidence score, metadata, and audit actor ID fields.
- The database preserves source attribution and historical observations.
- Product comparison is based on products, generic compositions, equivalence groups, and product equivalence mappings.
- Price intelligence begins with append-only `product_prices` observations.

## DRAP Import Memory

- DRAP import code lives in `src/modules/drap/`.
- DRAP module recovery notes live in `src/modules/drap/README.md`.
- The import flow is raw import, validation, normalization, duplicate detection, upsert strategy, audit log, and import report.
- CSV and HTML table parsing are built in.
- Excel is supported through an injected parser until an Excel dependency is installed.
- Future API support expects JSON array payloads.
- Import batches, row items, and row errors are stored in `import_batches`, `import_batch_items`, and `import_errors`.
- The sample dataset is `src/modules/drap/samples/drap.sample.csv`.
- Imports write manufacturers, generics, products, product_compositions, and audit_logs.

## Online Pharmacy Source Framework Memory

- Generic source framework lives in `src/modules/sources/`.
- Worker scaffolds live in `src/workers/source-sync.worker.ts` and `src/workers/price-sync.worker.ts`.
- No Dawaai, Sehat, DVAGO, Servaid, or other provider-specific scraper logic has been implemented.
- Adapter contract methods are `initialize`, `fetchProducts`, `fetchProductDetails`, `fetchPrices`, `normalize`, `validate`, `save`, and `healthCheck`.
- Common matching utilities are `matchBrand`, `matchGeneric`, `matchStrength`, `matchDosageForm`, and `generateMedicineSignature`.
- Mock validation assets live in `src/modules/sources/testing/`.
- Provider and sync metadata tables are `source_providers`, `source_provider_configs`, `source_sync_jobs`, `source_sync_results`, and `source_health_logs`.
- Price intelligence foundation tables are `price_snapshots`, `price_change_events`, and `product_availability`.
- Framework documentation lives in `docs/ONLINE_PHARMACY_FRAMEWORK.md`.

## Price Intelligence Memory

- Price Intelligence Engine lives in `src/modules/price-intelligence/`.
- Main service facade is `PriceIntelligenceService`.
- Product statistics are calculated by `PriceComparisonService`.
- City aggregation is calculated by `CityPriceAnalyticsService`.
- Price changes are detected by `PriceChangeDetectorService`.
- Market signals, trends, and anomalies are handled by `PriceAnalyticsService`.
- Analytics output tables are `product_price_statistics`, `city_price_statistics`, `market_price_signals`, `price_anomalies`, and `price_trends`.
- Input source is historical `price_snapshots`; snapshots should not be overwritten.
- Documentation lives in `docs/PRICE_INTELLIGENCE_ENGINE.md`.

## Medicine Matching Memory

- Medicine Matching Engine lives in `src/modules/matching/`.
- Main service facade is `MatchingService`.
- Canonical signatures are generated by `SignatureGeneratorService`.
- Confidence calculation is handled by `ConfidenceEngineService`.
- Matching dimensions are brand, generic, strength, dosage form, manufacturer, pack size, registration number, and medicine signature.
- Match statuses are `matched`, `possible_match`, `needs_review`, and `unmatched`.
- Canonical identity tables are `canonical_products`, `canonical_product_aliases`, `product_matches`, `match_reviews`, and `matching_rules`.
- Duplicate detection covers brands, products, manufacturers, and signatures.
- Documentation lives in `docs/MEDICINE_MATCHING_ENGINE.md`.

## Search API Memory

- Search API Foundation lives in `src/modules/search/`.
- Main service facade is `SearchService`.
- Product search is handled by `ProductSearchService`.
- Generic search is handled by `GenericSearchService`.
- Autocomplete is handled by `AutocompleteService`.
- Alternative search is handled by `AlternativeSearchService`.
- Ranking is handled by `SearchRankingService`.
- Search database tables are `search_cache`, `search_popularity`, `search_suggestions`, and `search_synonyms`.
- API endpoint contracts are documented in `docs/API_SPECIFICATION.md`.
- Search architecture and recovery are documented in `docs/SEARCH_API_FOUNDATION.md`.

## Product Discovery Memory

- Product Discovery Engine lives in `src/modules/discovery/`.
- Main service facade is `DiscoveryService`.
- Candidate generation is handled by `CandidateGeneratorService`.
- Evidence capture is handled by `EvidenceCollectorService`.
- Review transitions are handled by `DiscoveryReviewService`.
- Discovery statuses are `new`, `collecting_evidence`, `needs_review`, `approved`, `rejected`, and `merged`.
- Discovery tables are `discovery_candidates`, `discovery_evidence`, `discovery_reviews`, `discovery_jobs`, and `discovery_rules`.
- Discovery confidence combines source confidence, matching confidence, and evidence confidence.
- Documentation lives in `docs/PRODUCT_DISCOVERY_ENGINE.md`.

## Backend Runtime Memory

- Backend Runtime Foundation is implemented as an executable NestJS application.
- Runtime bootstrap lives in `src/main.ts`; root composition lives in `src/app.module.ts`.
- `RuntimeFeatureModule` registers the DRAP, sources, price intelligence, matching, search, and discovery foundations without creating public controllers yet.
- Prisma integration lives in `src/database/prisma.module.ts` and `src/database/prisma.service.ts`.
- Database health is checked through Prisma with a lightweight `SELECT 1`.
- Environment validation lives in `src/config/env.validation.ts`; config composition lives in `src/config/configuration.ts`.
- Health endpoints are `/health`, `/health/database`, and `/health/application`.
- Health module lives in `src/modules/health/`.
- Security controls include Helmet, CORS, Nest validation pipe, and throttling.
- Observability includes structured logging, request tracing, request logging, error logging, and startup diagnostics.
- Docker, Docker Compose, and Railway deployment scaffolding are present.
- `RAILWAY_SETUP.md` captures the deployment steps and CLI commands for Railway.
- `npm.cmd run build` and `npm.cmd test` passed after dependency installation and Prisma client generation.
